Domain Name System

Overview

The Domain Name System (DNS) is a hierarchical, distributed naming system that translates human-readable domain names into numerical IP addresses. DNS serves as the Internet's directory service, enabling users to access resources using memorable names rather than numerical addresses. Without DNS, all network resources would need to be referenced by their IP addresses directly.

DNS operates as a globally distributed database, with no single point of control or failure. This distributed architecture provides resilience, scalability, and localized performance through caching mechanisms. DNS is critical infrastructure; its failure would render the Internet effectively unusable for most practical purposes.

Domain Name System (DNS)
A hierarchical and decentralized naming system for computers, services, or other resources connected to the Internet or a private network. DNS translates domain names to IP addresses and provides other information through defined record types.

DNS Hierarchy Structure

DNS organizes domain names in a hierarchical tree structure. The root of this tree is represented by a dot (.), though it is typically omitted in common usage. Below the root are Top-Level Domains (TLDs), followed by second-level domains, and so forth.

DNS HIERARCHICAL STRUCTURE: [ . ] (ROOT) | +----------+------------+------------+----------+ | | | | | [.com] [.org] [.net] [.gov] [.mil] | | | | | +----+----+ | | | | | | | | | | [google] [amazon] [...] [...] [...] [...] | +---+---+ | | [www] [mail] Fully Qualified Domain Name (FQDN): www.google.com. | | | | | | | +-- Root (implied) | | +---- TLD (Top-Level Domain) | +---------- Second-Level Domain +---------------- Subdomain/Host Each level is separated by a dot. Maximum total FQDN length: 253 characters Maximum label length: 63 characters

Domain Name Components

Component Description Example
Root Domain Top of hierarchy, represented by dot . (usually implicit)
Top-Level Domain (TLD) Highest administrative level below root .com, .org, .uk, .edu
Second-Level Domain Registered domain name google, amazon, example
Subdomain Division within a domain www, mail, api
Host Specific device identifier server01, web-prod-01

Types of Top-Level Domains

Type Description Examples
gTLD (Generic) General purpose, not country-specific .com, .org, .net, .info
ccTLD (Country Code) Two-letter codes for countries/territories .us, .uk, .jp, .de, .au
sTLD (Sponsored) Sponsored by specific organizations .edu, .gov, .mil, .museum
Infrastructure Reserved for infrastructure purposes .arpa (Address and Routing)
New gTLDs Expanded generic domains (post-2012) .app, .dev, .cloud, .io

DNS Server Types

Root Name Servers

Root name servers are the authoritative name servers at the top of the DNS hierarchy. They respond to queries for the root zone and return referrals to TLD servers. TLD Name Servers

TLD name servers are authoritative for all domains within a specific top-level domain. When a root server receives a query for a .com domain, it refers the resolver to the .com TLD servers. TLD servers then provide referrals to the authoritative name servers for specific second-level domains.

Authoritative Name Servers

Authoritative name servers hold the actual DNS records for a specific zone. When a query reaches an authoritative server, it provides the definitive answer rather than a referral. Organizations typically operate at least two authoritative servers for redundancy.

Recursive Resolvers

Recursive resolvers (also called caching resolvers or DNS forwarders) are the DNS servers that client devices query directly. They perform the iterative resolution process on behalf of clients, querying root, TLD, and authoritative servers as needed. They cache responses to improve performance for subsequent queries.

DNS Resolution Process

DNS resolution is the process of converting a domain name to its corresponding IP address. This process may involve multiple queries across different server types, though caching typically reduces the number of queries required.

DNS RESOLUTION PROCESS (for www.example.com): CLIENT RECURSIVE ROOT .COM TLD EXAMPLE.COM RESOLVER SERVER SERVER AUTH NS | | | | | | 1. Query: | | | | | www.example.com | | | | |------------------->| | | | | | | | | | | 2. Query root | | | | | for .com | | | | |---------------->| | | | | | | | | | 3. Referral to | | | | | .com TLD servers| | | | |<----------------| | | | | | | | | | 4. Query TLD | | | | | for example.com | | | | |----------------------------------->| | | | | | | | | 5. Referral to | | | | | example.com NS | | | | |<-----------------------------------| | | | | | | | | 6. Query auth | | | | | for www record | | | | |------------------------------------------------>| | | | | | | | 7. Answer: | | | | | 93.184.216.34 | | | | |<------------------------------------------------| | | | | | | 8. Cached answer: | | | | | 93.184.216.34 | | | | |<-------------------| | | | | | | | | Subsequent queries for the same domain are answered from cache.

Query Types

Type Description Usage
Recursive Query Demands a complete answer (IP or error) Client to resolver
Iterative Query Accepts referrals to other servers Resolver to authoritative servers
Non-recursive Query Returns cached data only, no recursion Cache checks

DNS Record Types

DNS stores information in Resource Records (RRs). Each record has a specific type indicating what kind of data it contains. Understanding common record types is essential for DNS configuration and troubleshooting.

Type Name Purpose Example Value
A Address Maps hostname to IPv4 address 192.168.1.100
AAAA IPv6 Address Maps hostname to IPv6 address 2001:db8::1
CNAME Canonical Name Alias pointing to another hostname www -> webserver.example.com
MX Mail Exchanger Specifies mail server for domain 10 mail.example.com
NS Name Server Authoritative nameserver for zone ns1.example.com
PTR Pointer Reverse DNS: IP to hostname host.example.com
SOA Start of Authority Zone parameters and primary NS (serial, refresh, retry, expire)
TXT Text Arbitrary text data SPF, DKIM, verification strings
SRV Service Service location information _ldap._tcp.example.com
CAA Certificate Authority Authorization Allowed CAs for domain certificates 0 issue "letsencrypt.org"

Record Structure

DNS RESOURCE RECORD FORMAT: +-------+------+-------+-----+--------+-------+ | NAME | TYPE | CLASS | TTL | RDLEN | RDATA | +-------+------+-------+-----+--------+-------+ Fields: NAME - Domain name this record applies to TYPE - Record type (A, AAAA, MX, etc.) CLASS - Protocol class (usually IN for Internet) TTL - Time-to-live in seconds (caching duration) RDLEN - Length of RDATA field RDATA - Record-specific data Example A Record: www.example.com. IN A 3600 93.184.216.34 | | | | | +-- Name | | | +-- IP Address (RDATA) | | +-------- TTL (1 hour) | +----------- Type +--------------- Class

TTL and Caching

The Time-To-Live (TTL) value specifies how long a resolver should cache a record before querying for fresh data. TTL values balance between reducing DNS query load (longer TTL) and enabling rapid changes (shorter TTL).

Common TTL Values
Duration Seconds Use Case
5 minutes 300 Frequent changes, failover scenarios
1 hour 3600 Moderately stable records
24 hours 86400 Stable records
1 week 604800 Very stable records (NS, MX)

Reverse DNS (rDNS)

Reverse DNS maps IP addresses back to hostnames, the opposite of standard (forward) DNS resolution. Reverse DNS uses PTR records within special domains: in-addr.arpa for IPv4 and ip6.arpa for IPv6.

REVERSE DNS LOOKUP (IPv4): Forward: www.example.com -> 192.168.1.100 Reverse: 192.168.1.100 -> www.example.com Reverse zone construction: IP Address: 192.168.1.100 Step 1: Reverse the octets 100.1.168.192 Step 2: Append in-addr.arpa 100.1.168.192.in-addr.arpa Step 3: Query for PTR record 100.1.168.192.in-addr.arpa. PTR www.example.com. Use Cases: - Email server verification (spam filtering) - Security logging and forensics - Service validation - Compliance requirements

DNS Zone Files

A zone file is a text file containing all DNS records for a specific domain or portion of the DNS namespace. Zone files follow a standardized format defined in RFC 1035.

EXAMPLE ZONE FILE (example.com): ; Zone file for example.com ; Semicolon indicates comment $TTL 86400 ; Default TTL (24 hours) $ORIGIN example.com. ; Base domain ; SOA Record - Start of Authority @ IN SOA ns1.example.com. admin.example.com. ( 2024010101 ; Serial number (YYYYMMDDNN) 7200 ; Refresh (2 hours) 3600 ; Retry (1 hour) 1209600 ; Expire (2 weeks) 86400 ; Minimum TTL (24 hours) ) ; Name Server Records IN NS ns1.example.com. IN NS ns2.example.com. ; Mail Exchanger Records IN MX 10 mail.example.com. IN MX 20 backup-mail.example.com. ; A Records (IPv4 Addresses) IN A 93.184.216.34 ; example.com www IN A 93.184.216.34 ; www.example.com mail IN A 93.184.216.35 ns1 IN A 93.184.216.36 ns2 IN A 93.184.216.37 ; AAAA Records (IPv6 Addresses) www IN AAAA 2606:2800:220:1:248:1893:25c8:1946 ; CNAME Records (Aliases) ftp IN CNAME www.example.com. webmail IN CNAME mail.example.com. ; TXT Records IN TXT "v=spf1 mx a ip4:93.184.216.0/24 -all"

DNS Security

DNS Vulnerabilities

DNS was designed in an era when security was not a primary concern. Several vulnerabilities exist in the original protocol:

Attack Description Impact
DNS Spoofing Injecting false DNS responses Traffic redirection to malicious servers
Cache Poisoning Corrupting resolver cache with false data Persistent misdirection of users
DNS Amplification Using DNS for DDoS amplification Overwhelming target with reflected traffic
Zone Transfer Leakage Unauthorized zone transfer (AXFR) Reconnaissance, data exposure
DNS Tunneling Encoding data in DNS queries Data exfiltration, C2 communication

DNSSEC

DNS Security Extensions (DNSSEC) add cryptographic authentication to DNS responses. DNSSEC uses digital signatures to verify that DNS data has not been tampered with and originates from the authoritative source.

DNSSEC Record Types
Record Purpose
RRSIG Contains cryptographic signature for RRset
DNSKEY Public key for zone signing
DS Delegation Signer - hash of child zone key
NSEC/NSEC3 Authenticated denial of existence

DNS over HTTPS (DoH) and DNS over TLS (DoT)

DoH and DoT encrypt DNS queries to prevent eavesdropping and manipulation. They address privacy concerns but introduce operational considerations for network monitoring and security controls.

Protocol Port Transport Characteristics
Traditional DNS 53 UDP/TCP Unencrypted, easily monitored
DoT 853 TLS over TCP Encrypted, dedicated port
DoH 443 HTTPS Encrypted, blends with web traffic

  • Distributed architecture provides resilience and performance
  • Caching reduces load but requires TTL management
  • DNSSEC provides authentication but not encryption
  • DoH/DoT provide privacy but complicate network security
  • Understanding DNS is essential for troubleshooting connectivity
